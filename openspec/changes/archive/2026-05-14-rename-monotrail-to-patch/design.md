## Context

The codebase uses "Monotrail" as a diagram type name throughout all layers: the Mermaid diagram keyword, the npm package name, the `mermaid.initialize()` config key, TypeScript type names, internal function names, grammar source files, compiled grammar output, SVG CSS class names, and all documentation. The rename touches every file in the project but makes no functional changes.

## Goals / Non-Goals

**Goals:**
- Remove every occurrence of "Monotrail" / "monotrail" from all code, config, docs, and comments
- Rename the npm package to `mermaid-patch`
- Change the Mermaid diagram keyword from `monotrail` to `patch`
- Change the `mermaid.initialize()` config group key from `monotrail` to `patch`
- Rename internal TypeScript types, functions, and files consistently

**Non-Goals:**
- Changing any runtime behaviour, layout logic, rendering, or features
- Providing a compatibility shim / alias for the old `monotrail` keyword
- Versioning strategy or npm publish (out of scope for this change)

## Decisions

### 1. No backward-compatibility alias for the diagram keyword

**Decision:** Delete the `monotrail` keyword outright; `patch` is the only accepted start token.

**Rationale:** This is an early-stage library without a stable public release. A compatibility layer would add dead code and make the rename incomplete. A clean cut is simpler.

**Alternatives considered:**
- Accept both `monotrail` and `patch` via an alternation in the grammar — rejected: adds grammar complexity and perpetuates the old name in the codebase.

### 2. Rename grammar files alongside source

**Decision:** Rename `src/grammar/monotrail.ne` → `src/grammar/patch.ne` and update the compiled output path in `scripts/compileGrammar.cjs` and the import in `src/parser.ts`.

**Rationale:** Keeping file names consistent with the new identity prevents confusion for future contributors and keeps `.gitignore` comments accurate.

### 3. Rename internal TypeScript symbols

**Decision:** Rename `MonotrailAST` → `PatchAST`, `MonotrailConfig` → `PatchConfig`, `MonotrailDB` → `PatchDB`, `parseMonotrail` → `parsePatch`, and the exported constant `monotrail` → `patch` in `detector.ts`.

**Rationale:** Internal names should match the public-facing brand. Because these are not part of any published API surface (no `.d.ts` shipping), there is no external breakage beyond the package rename.

### 4. Rename SVG CSS class names

**Decision:** Rename `monotrail-edges`, `monotrail-badges`, `monotrail-nodes`, `monotrail-warnings` → `patch-edges`, `patch-badges`, `patch-nodes`, `patch-warnings`.

**Rationale:** These class names are part of the rendered SVG output. Users who write custom CSS targeting these classes will be affected, but since the class names carry the brand name they must change. This is documented as a breaking change.

## Risks / Trade-offs

- **User diagram breakage** → Mitigation: clearly document the breaking change in the changelog; the change is mechanical (global find-and-replace of the first line from `monotrail` to `patch`).
- **Custom CSS breakage** → Mitigation: document the SVG class rename; users targeting `.monotrail-*` classes must update their stylesheets.
- **Missed occurrences** → Mitigation: run a workspace-wide case-insensitive search for "monotrail" after implementation and confirm zero matches outside git history.
- **Compiled grammar is auto-generated** → The file `src/grammar/patch.grammar.js` is re-generated by `scripts/compileGrammar.cjs`; the old compiled file `src/grammar/monotrail.grammar.js` must be deleted and the script re-run to produce the new file.
